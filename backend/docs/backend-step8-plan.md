# STEP 8 계획 — 비동기·메시징 구조

## 1. 단계 목표

- **무거운 작업**(동기화, 집계, 알림, 외부 API 후처리)을 동기 HTTP 요청에서 **분리하는 기준**을 문서로 고정한다.
- **이벤트 형식**, **실패·재시도·멱등성** 정책을 정의해 구현 시 일관되게 적용한다.
- 초기에는 **브로커 없이**도 동작 가능한 구조를 두고, **RabbitMQ / Kafka / Redis Stream** 로의 확장 시 **포트·메시지 계약**이 유지되도록 검토한다.
- **로깅**(`requestId`, `syncJobId`)은 `backend-logging-spec.md`와 정합한다.

## 2. 이번 단계에서 해결할 문제

- 모든 것을 `@Async`로 처리하다 **스레드 풀 고갈**·**추적 불가**가 되는 문제.
- 동일 작업이 중복 실행되어 **YouTube 쿼터**만 소모하는 문제.
- 실패 시 **무한 재시도** 또는 **조용한 유실** 문제.
- 메시지 브로커 도입 시 **도메인 로직 전면 수정**이 필요해지는 문제.

## 3. 설계 대상

| 대상 | 설명 |
|------|------|
| 처리 대상 목록 | 동기화, 통계, 알림, 후처리 등 |
| 이벤트 envelope | 타입, 버전, payload, 상관 ID |
| 작업 저장소 | `sync_jobs` 상태 머신 |
| 재시도 | 횟수, 백오프, 재시도 불가 오류 |
| 멱등성 | 키 설계, 외부 API 기준 |
| 확장 | 인프라 추상화, 브로커별 어댑터 |
| 로그 | MDC, `jobId`/`syncJobId` |

## 4. 주요 결정 사항

- **MVP**: 메시지 브로커 **미도입**. **Spring `@Async`** + **DB `sync_jobs`** 로 작업 이력·상태를 관리한다.
- **도메인 이벤트**: 애플리케이션 내부는 **불변 이벤트 DTO** + `ApplicationEventPublisher`(선택); 외부 발행은 추후 **동일 스키마**를 메시지 본문으로 사용.
- **멱등성**: `job_type` + **비즈니스 키**(예: `youtubeChannelId`)로 **진행 중·최근 성공** 작업과 중복 생성 방지(락 또는 유니크 제약은 구현 단계).
- **재시도**: 일시적 오류(네트워크, 5xx)만; 4xx·인증 오류는 **FAILED** 처리 후 수동/별도 플로우.
- **확장**: `OutboundEventPublisher` 인터페이스(가칭)로 **브로커 구현체 스왑** 가능하게 설계(문서상 계약만 정의, 코드는 후속).
